# Prisma Patterns — RecallStack

This skill governs every interaction with the PostgreSQL database through
Prisma in the RecallStack project. It exists because the schema has several
patterns that must be applied consistently to avoid data corruption, performance
issues, and broken counts.

---

## ABSOLUTE RULES

1. Never update denormalized counts outside of a Prisma transaction.
2. Never write raw SQL unless Prisma cannot express the query. If raw SQL is
   needed, ask the user before writing it.
3. Never skip migrations. Every schema change must have a corresponding
   migration file generated by `prisma migrate dev`.
4. Never call `prisma.topic.count()` or `prisma.note.count()` on the homepage
   or any listing page. Use the denormalized `topicsCount` and `notesCount`
   fields instead. That is the entire reason they exist.
5. Always present the query plan to the user before writing complex queries
   involving nested includes or aggregations.

## DENORMALIZED COUNT UPDATE PATTERN

RecallStack stores pre-calculated counts on Subject and Topic to avoid
expensive COUNT queries on high-traffic pages. These counts must be kept
accurate using atomic Prisma transactions.

### When a Topic is created under a Subject

```js
/*
 * Creating a topic increments the parent subject's topicsCount by 1.
 * This is done inside a transaction so both writes succeed or both fail.
 * If they ran separately, a crash between the two writes would produce
 * an incorrect count with no way to detect it.
 */
await prisma.$transaction([
  prisma.topic.create({
    data: {
      name,
      slug,
      description,
      subjectId,
      order,
    },
  }),
  prisma.subject.update({
    where: { id: subjectId },
    data: { topicsCount: { increment: 1 } },
  }),
]);
```

### When a Topic is deleted

```js
/*
 * Deleting a topic decrements the parent subject's topicsCount.
 * The cascade delete on the schema handles notes and sections under this topic,
 * but the denormalized count on subject must be decremented manually here
 * because Prisma cascade logic does not trigger application-level transactions.
 *
 * Also decrement subject's notesCount by the number of notes this topic held.
 */
const topic = await prisma.topic.findUnique({
  where: { id: topicId },
  select: { notesCount: true, subjectId: true },
});

await prisma.$transaction([
  prisma.topic.delete({ where: { id: topicId } }),
  prisma.subject.update({
    where: { id: topic.subjectId },
    data: {
      topicsCount: { decrement: 1 },
      notesCount: { decrement: topic.notesCount },
    },
  }),
]);
```

### When a Note is published under a Topic

```js
/*
 * Publishing a note (status changes from DRAFT to PUBLISHED) increments
 * notesCount on both the parent topic and the grandparent subject.
 * lastUpdated on the topic is also refreshed.
 *
 * The note's topicId is required. The topic's subjectId is fetched first
 * so the transaction has both IDs available.
 */
const topic = await prisma.topic.findUnique({
  where: { id: topicId },
  select: { subjectId: true },
});

await prisma.$transaction([
  prisma.note.update({
    where: { id: noteId },
    data: { status: 'PUBLISHED', publishedAt: new Date() },
  }),
  prisma.topic.update({
    where: { id: topicId },
    data: {
      notesCount: { increment: 1 },
      lastUpdated: new Date(),
    },
  }),
  prisma.subject.update({
    where: { id: topic.subjectId },
    data: { notesCount: { increment: 1 } },
  }),
]);
```

### When a Note is deleted or archived

```js
/*
 * Only decrement counts if the note was PUBLISHED.
 * DRAFT notes never contributed to the counts so they must not decrement them.
 */
const note = await prisma.note.findUnique({
  where: { id: noteId },
  select: { status: true, topicId: true, topic: { select: { subjectId: true } } },
});

const wasPublished = note.status === 'PUBLISHED';

await prisma.$transaction([
  prisma.note.delete({ where: { id: noteId } }),
  ...(wasPublished
    ? [
        prisma.topic.update({
          where: { id: note.topicId },
          data: { notesCount: { decrement: 1 } },
        }),
        prisma.subject.update({
          where: { id: note.topic.subjectId },
          data: { notesCount: { decrement: 1 } },
        }),
      ]
    : []),
]);
```

## MIGRATION CONVENTIONS

Every schema change requires a migration. Follow these rules:

```bash
# Development: generates a migration file and applies it
npx prisma migrate dev --name <descriptive-name>

# Production: applies pending migrations without generating new ones
npx prisma migrate deploy
```

Migration name format: `snake_case`, describes the change, past tense.

```
# Good
npx prisma migrate dev --name add_seo_fields_to_note
npx prisma migrate dev --name create_analytics_table
npx prisma migrate dev --name add_difficulty_enum_to_note

# Bad (vague, no information)
npx prisma migrate dev --name update
npx prisma migrate dev --name fix
```

Never edit a migration file after it has been committed and applied.
If a migration was wrong, create a new migration to correct it.

After every migration, regenerate the Prisma client:
```bash
npx prisma generate
```

## AVOIDING N+1 QUERIES

N+1 happens when a loop fetches related data one record at a time instead of
joining in a single query. Always use `include` or `select` to load relations
in one query.

### Wrong — N+1 pattern

```js
// This runs one query per subject to get its topics count.
// With 20 subjects this is 21 queries.
const subjects = await prisma.subject.findMany();
for (const subject of subjects) {
  subject.topicsList = await prisma.topic.findMany({
    where: { subjectId: subject.id },
  });
}
```

### Correct — single query with include

```js
/*
 * Include fetches topics in the same query using a JOIN.
 * For the homepage, we do not need topics at all — the denormalized
 * topicsCount field on subject is sufficient. No include needed.
 */
const subjects = await prisma.subject.findMany({
  orderBy: { order: 'asc' },
  // topicsCount and notesCount are already on the subject row.
  // No include required for the homepage cards.
});
```

### When you need nested data

```js
/*
 * Fetching a topic page: need the subject name for breadcrumbs,
 * and notes with their section count for the note list.
 * select is used instead of include to avoid over-fetching.
 */
const topic = await prisma.topic.findUnique({
  where: { slug: topicSlug, subject: { slug: subjectSlug } },
  select: {
    id: true,
    name: true,
    description: true,
    notesCount: true,
    lastUpdated: true,
    subject: {
      select: { id: true, name: true, slug: true },
    },
    notes: {
      where: { status: 'PUBLISHED' },
      orderBy: { publishedAt: 'desc' },
      select: {
        id: true,
        title: true,
        slug: true,
        excerpt: true,
        difficulty: true,
        readingTime: true,
        tags: true,
        publishedAt: true,
        _count: { select: { sections: true } },
      },
    },
  },
});
```

## READING TIME CALCULATION

Reading time is stored on the Note as an integer (minutes). It is calculated
from the combined character count of all sections when a section is saved.

```js
/*
 * Recalculate reading time after any section is created, updated, or deleted.
 * Average reading speed: 200 words per minute.
 * Average word length in technical content: 5 characters.
 * Formula: total characters / 5 / 200, rounded up to nearest minute, minimum 1.
 */
async function recalculateReadingTime(noteId) {
  const sections = await prisma.section.findMany({
    where: { noteId },
    select: { content: true },
  });

  const totalChars = sections.reduce((sum, s) => sum + s.content.length, 0);
  const minutes = Math.max(1, Math.ceil(totalChars / 5 / 200));

  await prisma.note.update({
    where: { id: noteId },
    data: { readingTime: minutes },
  });
}
```

## SECTION REORDER PATTERN

Sections have an `order` integer field. Reordering must update all affected
sections in a single transaction to prevent order collisions.

```js
/*
 * sectionOrders is an array like: [{ id: 'abc', order: 0 }, { id: 'xyz', order: 1 }]
 * All sections in the array belong to the same note.
 * The transaction updates each section's order atomically.
 */
async function reorderSections(sectionOrders) {
  await prisma.$transaction(
    sectionOrders.map(({ id, order }) =>
      prisma.section.update({ where: { id }, data: { order } })
    )
  );
}
```

## SLUG UNIQUENESS RULES

- Subject slugs are globally unique (`@@unique` on slug).
- Topic slugs are unique within a subject (`@@unique([subjectId, slug])`).
- Note slugs are unique within a topic. Add this constraint if not already
  in the schema, and confirm with the user before running the migration.

Slug generation always uses the `slugify` utility from `backend/src/utils/slugify.js`.
Never generate slugs manually inline. Never allow the user to set a slug
directly through the API without validation.

## SEED FILE CONVENTIONS

The seed file lives at `prisma/seed.js`. It creates a minimal but realistic
dataset for local development.

```js
/*
 * FILE: prisma/seed.js
 *
 * PURPOSE:
 * Seeds the database with a minimal realistic dataset for local development.
 * Creates one admin user, three subjects (DSA, System Design, Interview Prep),
 * two topics per subject, and two published notes per topic with sections.
 *
 * Run with: npx prisma db seed
 * This script is idempotent: running it twice does not create duplicates
 * because upsert is used instead of create.
 */
```

Always use `upsert` in seeds so they are safe to re-run:

```js
await prisma.subject.upsert({
  where: { slug: 'dsa' },
  update: {},
  create: {
    name: 'DSA',
    slug: 'dsa',
    description: 'Data Structures and Algorithms',
    icon: 'chart',
    color: '#3B82F6',
    order: 0,
    topicsCount: 0,
    notesCount: 0,
  },
});
```

## PRISMA CLIENT SINGLETON

In the Next.js frontend, Prisma client must be instantiated as a singleton
to avoid exhausting the connection pool during hot reload in development.

```js
/*
 * FILE: frontend/src/lib/prisma.js
 *
 * PURPOSE:
 * Exports a single shared Prisma client instance.
 * In development, Next.js hot reload creates new module instances on each
 * reload. Without this singleton pattern, each reload would open a new
 * database connection, exhausting the pool quickly.
 * The global variable persists across hot reloads in development only.
 */

import { PrismaClient } from '@prisma/client';

const globalForPrisma = globalThis;

export const prisma =
  globalForPrisma.prisma ?? new PrismaClient({ log: ['error'] });

if (process.env.NODE_ENV !== 'production') {
  globalForPrisma.prisma = prisma;
}
```

The backend uses a separate instance in `backend/src/config/database.js`
with the same singleton pattern.
